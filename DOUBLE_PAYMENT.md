# Защита от двойной оплаты (Double Payment Protection)

Этот документ описывает реализацию механизма защиты от двойных списаний (Double Payment Protection) при оплате брони в слотах.

##  Краткое описание
Реализована многоуровневая защита, которая гарантирует, что пользователь не сможет случайно или намеренно оплатить участие в одном и том же слоте дважды. Система устойчива к сетевым дублям (retries), параллельным запросам (race conditions) и программным сбоям.

##  Как это работает (Уровни защиты)

1. **Идемпотентность (Idempotency Key)**
   - API требует передачи HTTP-заголовка `X-Idempotency-Key` (желательно UUID).
   - При успешной оплате ключ сохраняется в Redis на 24 часа.
   - Если клиент повторяет запрос из-за таймаута сети с *тем же ключом*, сервер сразу возвращает сохраненный успешный статус (`200 OK`), не трогая БД и биллинг.

2. **Распределенная блокировка (Redis Lock)**
   - На время обработки платежа берется мьютекс в Redis по ключу: `payment:participant:{userId}:{slotId}`.
   - Если придут два одновременных запроса с *разными* ключами идемпотентности, второй запрос будет заблокирован и дождется завершения первого, после чего получит ошибку `ALREADY_PAID`.

3. **Транзакции базы данных (PostgreSQL)**
   - Смена статуса участника (`RESERVED` ➔ `PAID`) и создание записи чека о платеже в таблице `payments` происходят в единой транзакции (ACID). Если что-то падает, всё откатывается. 

4. **Защита на уровне схемы (DB Constraints)**
   - В таблице `payments` стоит строгое ограничение `UNIQUE(participant_id)`.
   - Даже если откажут мьютексы и логика в коде, сама СУБД физически не позволит вставить два платежа для одного участия.

##  Как тестировать

### 1. Подготовка окружения
Пересоберите и запустите базу данных и API в фоновом режиме:
```powershell
docker compose down -v
docker compose up -d --build
```
*(Базе данных нужно около 3-5 секунд на применение миграций).*

### 2. Запуск E2E автотеста
В корне проекта выполните специальный тестовый скрипт:
```powershell
.\proof.ps1
```

### 3. Ожидаемый результат
Вы должны увидеть последовательное прохождение флоу:
1. Запись в слот ➔ `success: true`
2. Нормальная оплата с Ключом А ➔ `success: true`
3. Повторная отправка запроса с Ключом А (симуляция лага сети) ➔ `success: true, message: Payment already processed` (Ответ из кэша)
4. Попытка двойной оплаты с новым Ключом Б ➔ `error: already_paid` (Блокировка сервером)
