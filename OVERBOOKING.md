# Защита от овербукинга 

В этом проекте реализована надежная защита от овербукинга (переполнения слотов), чтобы пользователи не могли занять мест больше, чем позволяет вместимость (capacity) игрового слота.

## Что было сделано

Система защиты построена на 4 уровнях:
1. **Redis Distributed Lock** — блокировка на уровне кэша. Исключает "race conditions" (состояние гонки), когда несколько пользователей одновременно нажимают кнопку.
2. **PostgreSQL Transactions** — гарантирует атомарность (ACID). Если база упадет посреди записи, данные откатятся.
3. **Capacity Check** — перед записью БД считает уже существующих участников и сверяет с лимитом.
4. **UNIQUE Constraint** — ограничение в БД `(slot_id, user_id)`, которое физически не даст одному человеку занять два места в одном слоте.

## 📁 Где лежит код

Основная логика распределена по двум компонентам:
- **Обработчик API и транзакции:** `apps/api/internal/handlers/routes.go` (функция `Join`)
- **Распределенная блокировка Redis:** `apps/api/internal/locking/redis.go`

## Как проверить

У вас есть готовый скрипт, который имитирует одновременное подключение множества пользователей к одному слоту.

1. **Запустите проект:**
   ```bash
   docker compose up --build -d
   ```

2. **Запустите тестовый скрипт в самом проекте:**
   Скрипт попытается "засунуть" 8+ человек в один слот.
   ```powershell
   .\run-test.ps1
   ```

3. **Ожидаемый результат:**
   ```text
   Slot capacity: 8, Current: 0
   [OK] 2222... - Joined slot (1/8 participants)
   ...
   [OK] 8888... - Joined slot (8/8 participants)
   [FAIL] 9999... - slot_full
   
   Results: Joined=8 Failed=1 Final=8/8
   SUCCESS: Overbooking protection is working!
   ```
   Лишние пользователи получат ошибку `slot_full` с кодом HTTP **409 Conflict**.
